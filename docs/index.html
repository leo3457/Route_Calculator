1. Introduction

This case study outlines the end-to-end data science and engineering workflow for processing high-frequency Electric Vehicle (EV) CAN bus telemetry. The primary goal of this pipeline is to ingest raw “golden signals” (GPS, odometer, and battery state-of-charge), generate deterministic driving routes, and visualize them for downstream predictive maintenance using deep learning models.

2. Data Acquisition (Import)

Raw telemetry data is continuously emitted by the vehicle’s IoT gateway. To extract this data, I utilized specific SQL extraction scripts (route_queries.py) to pull time-series data from our backend databases. The raw datasets contain critical metrics: * time: UTC Timestamps. * odometer: Cumulative distance tracking. * soc: Battery State of Charge (%). * ignition: Binary status of the vehicle engine.

To ensure reproducible testing during development, I took snapshots of this data, resulting in files like sample_golden_signal_opent1.csv.

3. Data Transformation & The Processing Engine

The core of this project is the Route Calculator Engine. The engine parses the continuous stream of CAN bus data and splits it into discrete “trips” or “routes” based on specific logic (e.g., when the vehicle charges vs. when it is driving).

The Performance Bottleneck (route_calculator.py)

Initially, the engine processed the dataframe using Pandas’ df.iterrows(). ```python # Old Implementation (Slow) for _, row in df.iterrows(): odo_mi = row[‘odometer’] * 0.000621371 time_gap = (row[‘time’] - prev_time).total_seconds()

This approach created severe CPU bottlenecks. Iterating row-by-row in Python meant the interpreter had to perform type checks, allocate memory, and dispatch operations over 500,000+ times per run. Processing a standard batch of 100 trucks took ~48 to 50 seconds.

The Refactor (route_calculator2.py) To achieve production-ready latency, I completely refactored the calculation loop to leverage Pandas vectorization and itertuples(). By treating columns as contiguous memory blocks rather than iterating through individual Python Series objects, the processing time dropped to ~0.7 seconds.

Result: A 64x speed improvement, enabling the API to return responses almost instantly.

Validation and Testing Data integrity is critical when refactoring data science pipelines. I created comprehensive unit tests (test_route_cal.py and test_route_cal2.py) comparing the output of the old engine against the new engine using our sample CSV datasets to ensure 100% parity in the resulting deterministic driving routes.

  1. Visualization & Communication With the data efficiently transformed, it is exposed to our Grafana Service Dashboard to provide immediate insights to stakeholders.

Interactive Mapping: The driving_route table features dynamic Data Links. By clicking on a specific start or end time, the Grafana URL parameters (from and to) are automatically updated.

Timezone Alignment: A critical hurdle was aligning the UTC timestamps generated by the backend with the local browser time (PST). By standardizing the dashboard’s global time settings to UTC and pushing raw millisecond epochs into the URL variables, the GPS map accurately queries and displays only the telemetry for that exact 50-mile or 86-mile drive.

Efficiency Metrics: The table automatically calculates kWh/mile allowing technicians to spot battery degradation at a glance.

  1. Modeling (Next Steps) The successful transformation and speed optimization of the Route Calculator lays the groundwork for advanced predictive analytics. Because the route boundaries are now cleanly separated and calculated in sub-seconds, we can efficiently feed these route sequences into an Informer Model.

Unlike standard Transformers that suffer from quadratic complexity, the Informer’s ProbSparse self-attention mechanism is perfectly suited to handle the long-sequence time-series data of our EV telemetry, allowing us to predict battery faults before they occur.
